# v1.2.1 Milestone — Live-Smoke Fixes & Self-Host Password UX

**Initiated:** 2026-04-24 — immediately after v1.2.0 shipped to GHCR.
**Trigger:** Full-journey Playwright live-smoke test run against the VPS deployment uncovered 2 production bugs that no unit or integration test had exercised. A 3rd dead-link bug + a 4th UX concern (password policy mismatch between login and signup) surfaced during the fix round.

## Executive Summary

v1.2.0 shipped with cosign, SBOM, SLSA-3 provenance, and 672/672 tests green — but the first real end-to-end user journey on a live HTTP deploy found 2 real failures that the hermetic test suite couldn't see. Plus, enabling the `/deployment` learn-more link on the insecure-context banner was pointing at a route that never existed, and the tightened 12-char password bar from Phase 23 SEC-06 clashed with a legacy 8-char login-schema floor — producing confusing messages for self-hosted users.

This patch milestone closes the feedback loop by:
1. Restoring cookie delivery on LAN-HTTP production deploys
2. Routing Next.js API paths correctly behind Caddy
3. Removing the dead `/deployment` link
4. Introducing `PASSWORD_POLICY=simple|strong` with `simple` as the default (8-char floor everywhere) — self-hosted operators get an unopinionated baseline; public-facing operators opt into `strong` (12-char signups) explicitly.

## Phase Plan

### Phase 29 — Auth Cookie + Remember Me *(retroactive)*

- **Covers:** PATCH2-01, PATCH2-02
- **Commit:** `3215ca8`
- **Files:** `lib/actions/auth.ts`, `components/forms/login-form.tsx`
- **Problem:** `secure: process.env.NODE_ENV === 'production'` set `Secure=true` on the VPS (`SITE_URL=http://46.62.151.57:3000`, `NODE_ENV=production`). Browsers drop Secure cookies over HTTP → the auth cookie was invisible → signup succeeded but every authed navigation bounced to `/login`. Fix derives `Secure` from `SITE_URL.startsWith('https://')`. Bundled a Remember-me checkbox (default checked = 14-day persistent; unchecked = session cookie) in the same commit because both touch the same `cookieOptions()` helper.
- **Verification:** v1.2 live smoke passes; session-persistence assertion (`/login` → `/h` redirect while cookie present) satisfied; hard-logout via `ctx.clearCookies()` + re-login restores identical home.

### Phase 30 — Caddy Next.js API Routing *(retroactive)*

- **Covers:** PATCH2-03
- **Commit:** `731acc5`
- **Files:** `docker/Caddyfile`, `tests/e2e/v1.2-live-smoke.spec.ts`
- **Problem:** `handle /api/* { reverse_proxy localhost:8090 }` (PocketBase) was the only matcher in the `/api/*` family. Next.js API routes (`/api/health`, `/api/csp-report`, `/api/admin/*`, `/api/demo/*`) fell through to PB and 404'd. Fix installs an explicit allow-list ordered before the PB catch-all.
- **Verification:** `curl -sI -X POST http://127.0.0.1:3000/api/csp-report | head -1` returns `204 No Content`; PB collection routes (`/api/collections/*`) still reach PB; `/_/` still 404 via the admin-block matcher.

### Phase 31 — Fix `/deployment` Dead Link *(new)*

- **Covers:** PATCH2-04
- **File:** `components/insecure-context-banner.tsx`
- **Problem:** `InsecureContextBanner` renders `<Link href="/deployment">Learn more</Link>` — Next.js 16 auto-prefetches the RSC payload on hydration. `/deployment` is not a route. Every HTTP visitor sees 9+ console 404s (one per page visit that renders the banner — which is every page on HTTP). Not a functional bug but noisy for operators watching browser devtools.
- **Fix:** Drop the link. Banner keeps the informational copy ("install-to-home-screen and offline support require HTTPS") — still dismissible, still accurate. Full deployment guidance lives in `docs/deployment-hardening.md` which self-hosters reach from the repo.
- **Verification:** Re-run the 404 profiler; zero `/deployment` hits.

### Phase 32 — Configurable Password Policy *(new)*

- **Covers:** PATCH2-05
- **Files:** `lib/schemas/auth.ts`, `.env.example`, `docs/deployment-hardening.md`
- **Problem:** Phase 23 SEC-06 raised signup/reset-confirm min length 8 → 12 for public-facing deployment safety. Login-schema stayed at 8 for back-compat with pre-SEC-06 accounts. Self-hosted users running on a LAN see **two** different error messages during the same flow (signup says "at least 12", login says "at least 8"). The 12-char bar is also excessive for a single-household self-host with no public surface.
- **Fix:** Add `PASSWORD_POLICY` env var.
  - `simple` (default) — 8-char floor on signup, login, reset. Appropriate for self-hosted + LAN + Tailscale deployments where the threat model doesn't include credential stuffing.
  - `strong` — 12-char floor on signup + reset (Phase 23 SEC-06 behavior). Login keeps the 8-char floor so pre-flip accounts aren't locked out.
  - `docs/deployment-hardening.md` adds a "Set `PASSWORD_POLICY=strong` for public deploys" item.
- **Verification:** Unit tests cover both modes. Live-smoke signup with a 10-char password succeeds under default `simple`.
