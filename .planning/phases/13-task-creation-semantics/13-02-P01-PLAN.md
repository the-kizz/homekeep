## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client form → createTask (FormData serialization) | last_done field is user-controlled date string; schema validates; server treats empty as null |
| Form Controller → RHF state | Date input may emit partial / malformed strings mid-typing; zod nullable().optional() tolerates; only finalized submit reaches server |
| Integration-test disposable PB → host filesystem | DATA_DIR cleanup via rmSync in beforeAll+afterAll — no persistent state between test runs |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-05 | Tampering | Malformed last_done (e.g. "2026-99-99", "not-a-date") reaches createTask | mitigate | `new Date(parsed.data.last_done)` returns Invalid Date for malformed input; addDays throws NaN-guard would propagate. Wave 1's D-06 placement error catch swallows and falls back to null next_due_smoothed. Tests would benefit from an explicit "malformed last_done → natural fallback" unit test, but D-17 test scope is 12 unit + 3 integration — defense-in-depth via D-06 is sufficient. |
| T-13-06 | Information Disclosure | last_done field visible for anchored tasks could leak "user intended cycle, reverted to anchored" workflow signal | accept | The form hides the field when scheduleMode !== 'cycle' — no disclosure risk. If user toggles anchored after entering a last_done, RHF state retains it but the field is not rendered and not submitted (Controller is outside the conditional render, so the input is not mounted to the DOM on anchored). Belt-and-braces: server ignores last_done when schedule_mode === 'anchored' (computeFirstIdealDate throws on anchored; createTask's Wave 1 block gates on cycle). |
| T-13-07 | Denial of Service | Port 18101 collision with another local dev PB or test suite | mitigate | Phase 11 + Phase 12 established the port register (18090..18100). Plan claims 18101 via grep-unique constant + inline allocation comment. If a dev runs two parallel test suites, vitest pool concurrency would race — current project convention is single-file test runs for disposable-PB tests (observable in Phase 11/12 SUMMARYs). |
</threat_model>
