<objective>
Schema + hooks + rule-migration + backfill. This is the foundational Wave that every other Phase 4 plan depends on. Three PocketBase migrations land in strict timestamp order: `1714953600_home_members.js` creates the home_members collection AND backfills all existing homes with an owner membership row; `1714953601_invites.js` creates the invites collection (token unique, owner-only API rules); `1714953602_update_rules_multi_member.js` swaps homes/areas/tasks/completions list/view/create/update rules from owner-only to member-of-home using the PB 0.22+ `_via_` back-relation with the `?=` any-match operator. The existing `pocketbase/pb_hooks/homes_whole_home.pb.js` hook is extended (not forked) so new home creation atomically inserts owner → home_members in the same DB transaction as the Whole Home area.

Purpose: Multi-user from v1.0 onward requires membership to be the DB-enforced gate, not a server-action preflight. By landing the rules migration in Wave 1, Wave 2's server actions are free to rely on PB as the single source of ownership truth — with server-side `assertMembership` as defense-in-depth, not primary gate. The backfill + hook extension ensure both the 2.1-deployed production homes and all future homes carry their owner membership atomically — no orphan homes, no owner who can't read their own home post-migration (Pitfall 1).

Output: Two new PB collections (home_members, invites), one rewritten hook, three migration files, two disposable-PB integration tests on port 18092 proving atomicity + backfill + rule isolation.
</objective>
