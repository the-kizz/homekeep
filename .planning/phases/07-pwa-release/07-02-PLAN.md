    2. **docker/docker-compose.caddy.yml** (overlay, compose-merge with docker-compose.yml):
       ```yaml
       # HTTPS variant: add a Caddy sidecar in front of the homekeep service.
       # Usage: docker compose -f docker/docker-compose.yml -f docker/docker-compose.caddy.yml up -d
       # Requires: DOMAIN in .env, A-record DOMAIN -> VPS public IP, ports 80+443 open.

       services:
         homekeep:
           # Hide the host port from the overlay -- Caddy reaches it over the compose network.
           ports: !reset []

         caddy:
           image: caddy:2.11.2-alpine
           container_name: homekeep-caddy
           restart: unless-stopped
           depends_on:
             - homekeep
           ports:
             - "80:80"
             - "443:443"
             - "443:443/udp"   # HTTP/3 (Caddy 2.6+)
           volumes:
             - ./Caddyfile.prod:/etc/caddy/Caddyfile:ro
             - caddy_data:/data
             - caddy_config:/config
           environment:
             - DOMAIN=${DOMAIN:?set DOMAIN in .env before using this variant}
             - CADDY_EMAIL=${CADDY_EMAIL:-admin@${DOMAIN}}

       volumes:
         caddy_data:
         caddy_config:
       ```
       Key design points:
       - Image pinned to `caddy:2.11.2-alpine` -- same tag the Dockerfile uses for the internal Caddy binary (01-02 SUMMARY). Supply-chain coherence.
       - `!reset []` removes the public port mapping from the base service so only Caddy binds to host ports (D-11: compose-merge strategy). Requires Docker Compose v2.24+. Fallback for older compose: use `ports: []` (behaviour identical since compose merges lists on older versions).
       - `${DOMAIN:?...}` fails fast with a clear error if DOMAIN is unset -- better than Caddy silently serving a localhost cert.
       - Named volumes `caddy_data` + `caddy_config` persist the Let's Encrypt cert and ACME account key across restarts. CRITICAL: without these the rate limits (5 certs/week/domain) kick in fast on restart loops.
       - HTTP/3 on UDP 443 is a free perf win on Caddy 2.6+.

    3. **docker/docker-compose.tailscale.yml** (overlay):
       ```yaml
       # Tailscale variant: add a tailscale sidecar that puts homekeep on the tailnet.
       # Usage: docker compose -f docker/docker-compose.yml -f docker/docker-compose.tailscale.yml up -d
       # Requires: TS_AUTHKEY in .env (reusable auth key from https://login.tailscale.com/admin/settings/keys)
       # MagicDNS + HTTPS must be enabled on the tailnet for install-as-PWA to work.

       services:
         homekeep:
           # Share the tailscale sidecar's network namespace so the tailscale IP fronts the app.
           network_mode: "service:tailscale"
           ports: !reset []

         tailscale:
           image: tailscale/tailscale:stable
           container_name: homekeep-tailscale
           restart: unless-stopped
           hostname: homekeep
           environment:
             - TS_AUTHKEY=${TS_AUTHKEY:?set TS_AUTHKEY in .env before using this variant}
             - TS_STATE_DIR=/var/lib/tailscale
             - TS_USERSPACE=false
             - TS_EXTRA_ARGS=--advertise-tags=tag:container
             - TS_SERVE_CONFIG=/config/serve.json
           volumes:
             - tailscale_state:/var/lib/tailscale
             - /dev/net/tun:/dev/net/tun
           cap_add:
             - net_admin
             - sys_module

       volumes:
         tailscale_state:
       ```
       Key design points:
       - `network_mode: "service:tailscale"` shares the sidecar's net namespace with homekeep so the tailscale IP fronts the app (D-10 -- standard Tailscale Docker pattern).
       - Pinned to `tailscale/tailscale:stable` -- Tailscale publishes `stable` as a floating tag. SHA-pinning tailscale is deferred to post-v1 if needed (non-blocking).
       - `TS_USERSPACE=false` + `net_admin` + `/dev/net/tun` = kernel networking (faster than userspace wireguard-go). Requires host kernel wireguard, which every modern Linux kernel has. Fallback: set `TS_USERSPACE=true` and remove cap_add/volume on hosts without /dev/net/tun.
       - `TS_SERVE_CONFIG` gets set even though we do not ship a serve.json yet; advanced operators can drop one in to enable funnel (public HTTPS). Basic tailnet-internal access works without it.
       - `TS_AUTHKEY` uses `:?` same as DOMAIN for fail-fast clarity.

       # Installing HomeKeep as a PWA

       Once HomeKeep is served over HTTPS (Caddy or Tailscale variant), it can be installed as a Progressive Web App on phones, tablets, and desktops -- giving you a home-screen icon and a full-screen window without browser chrome.

       1. Open HomeKeep in Safari.
       2. Tap the Share button (square with an up arrow).
       3. Scroll and tap **Add to Home Screen**.

       ## Android (Chrome)
       1. Open HomeKeep in Chrome.
       2. Tap the three-dot menu.
       3. Tap **Install app** (or **Add to Home screen**).
       4. Confirm; the icon lands on your home screen and launcher.
       Chrome will usually surface an install prompt automatically on second visit.

       ## Desktop (Chrome, Edge, Brave)
       1. Open HomeKeep in the browser.
       2. Look for the install icon in the address bar (rectangle-with-down-arrow, right side).
       3. Click **Install**.
       The app opens in its own window, pinnable to the taskbar or dock.

       ## What works offline
       Once installed and visited at least once online, HomeKeep caches the dashboard shell and falls back to an offline page when the network is unreachable. Read-only actions (viewing tasks you have already loaded) work offline; completing tasks requires connectivity (writes do not queue in v1).

       ## Uninstalling
       - iOS: long-press icon -> Remove App -> Delete from Home Screen.
       - Android: long-press icon -> Uninstall.
       - Desktop Chrome: open the app, three-dot menu -> Uninstall HomeKeep.
