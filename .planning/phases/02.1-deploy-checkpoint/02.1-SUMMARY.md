2. Opened VPS firewall port 3000: `ufw allow 3000/tcp` — done (port 80 already held by aom-wiki-caddy, so we used 3000 for HomeKeep).

5. Started stack: `docker compose -f docker/docker-compose.yml -f docker/docker-compose.dev.yml up -d`.
6. Smoke-tested all four HTTP surfaces (landing, login, signup, health, PB admin) both locally (127.0.0.1:3000) and externally (46.62.151.57:3000).

## Redeploy command (for user)

When the user wants to refresh the deployed app after a git pull or code change:

```bash
cd /root/projects/homekeep && \
  docker buildx build --platform linux/amd64 --load -t homekeep:latest -f docker/Dockerfile . && \
  docker compose -f docker/docker-compose.yml -f docker/docker-compose.dev.yml up -d --force-recreate
```

One-liner saved in project memory + this SUMMARY. No data loss — named volume `docker_homekeep_data` persists across `up/down` cycles.

## Known gaps (accept for now)

- **arm64 cross-build still in-flight from Phase 1 re-verification** — running under QEMU in a background task. If it finishes before Phase 3 kicks off, the result will be logged; if not, CI will validate on every PR from Phase 3 onward.
- **Data named volume, not bind mount** — host-side `./data` isn't used in the VPS deploy because of the uid-remapping issue. The user-facing docs in README still reference `./data` as the recommended approach for operators whose host uid is 1000 (the common case); a VPS running Docker with userns-remap configured will always need a named volume.
- **Rate limits bumped for E2E** — `*:authWithPassword` is 20/60s (was 5/60s in CONTEXT). The tighter prod value returns in Phase 7.
- **SMTP not configured** — password-reset emails will silently no-op until `SMTP_HOST`/`SMTP_USER`/`SMTP_PASS` are set. Graceful-degradation per D-02 is active; the UI shows a reassuring message.
